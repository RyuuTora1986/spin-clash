# 功能变更分类规则

日期：2026-04-24

## 目的

以后任何“加功能、加系统、加玩法、改接入、改平台适配”的工作，都先做分类。  
**先分类，再动代码，再构建渠道包。**

如果不做这一步，最容易出现的问题是：

- 本来只该改 `core`，结果手工改了多个渠道
- 本来只是渠道包装差异，结果误改到母体
- 新功能上线后，某个旧渠道差异被不小心覆盖

## 规则总览

所有变更必须先归到以下 6 类中的 1 类。

### 1. `core_gameplay`

定义：

- 战斗规则
- 物理
- 胜负判定
- 核心成长结构
- 通用关卡规则
- 通用内容数据结构

处理规则：

- 先改内部母体
- 默认认为所有渠道都应继承
- 改完后必须跑：
  - `npm run verify:channels`

不该做的事：

- 不要直接在某个渠道版里单独改核心玩法

### 2. `shared_capability`

定义：

- 分析服务
- 存档能力
- 后端桥接
- 账号能力
- 分享能力
- 奖励能力抽象层

处理规则：

- 先改共享能力层
- 再看各渠道是否允许/启用/禁用
- 必须检查：
  - registry 是否要更新
  - build/check 是否要更新

### 3. `channel_wrapper`

定义：

- 首页 CTA
- 按钮文案
- 入口优先级
- 无广告渠道的替代引导
- 特定渠道的轻量经济包装

处理规则：

- 不改母体玩法真相
- 只改渠道 override / policy / adapter
- 必须补：
  - `behaviorContracts`
  - 对应 smoke/check

这是当前 `crazygames_basic` 的主要类型。

### 4. `platform_integration`

定义：

- 广告 SDK
- 平台账号
- 平台云存档
- 平台支付
- 提交包结构

处理规则：

- 先看平台规则
- 再改 registry、build、check、smoke
- 这类改动默认是高风险，不能只改业务代码

### 5. `content_only`

定义：

- 新场地
- 新陀螺
- 新节点
- 新文本
- 新素材

处理规则：

- 如果只是通用内容，优先走母体
- 只有平台硬规则要求时，才允许渠道专属内容
- 如增加了新可见路径或入口，仍要检查是否影响渠道行为契约

### 6. `ops_only`

定义：

- 构建脚本
- 发布脚本
- 检查脚本
- 提交素材脚本
- 文档和清单

处理规则：

- 不改玩法
- 但必须跑对应验证链

## 快速判断法

如果一个新改动满足下面任意一条，先按 `core_gameplay` 处理：

- 它改变了“这还是不是同一个游戏”
- 它影响胜负、节奏、成长主线
- 它不应该因为渠道不同而变成完全不同的规则

如果一个新改动满足下面任意一条，先按 `channel_wrapper` 处理：

- 它只是在改怎么引导
- 它只是在改怎么展示
- 它只是在补某个平台被禁止功能后的替代方案

如果一个新改动满足下面任意一条，先按 `platform_integration` 处理：

- 它依赖外部平台 SDK
- 它依赖平台登录、支付、广告、云存档
- 它不改构建/验证就无法安全上线

## 每类变更的最小动作

### `core_gameplay`

必须做：

1. 改母体
2. 判断是否冲击渠道行为契约
3. 跑 `npm run verify:channels`

### `shared_capability`

必须做：

1. 改共享层
2. 检查 registry
3. 检查对应渠道 check/smoke

### `channel_wrapper`

必须做：

1. 改渠道 override / wrapper
2. 把差异写进 `behaviorContracts`
3. 跑对应 `verify:channel`

### `platform_integration`

必须做：

1. 核对平台规则
2. 改 registry
3. 改 build/check/smoke
4. 跑对应渠道验证

## 禁止事项

### 禁止 1：为了图快，把渠道差异直接写进母体核心逻辑

除非该差异本来就是全渠道通用改进，否则不要这样做。

### 禁止 2：只改了渠道代码，但没有补验证

这类差异不算真正受保护。

### 禁止 3：不分类就直接开工

这样最容易让项目重新回到“靠记忆维护”的状态。

## 当前项目的典型例子

### `crazygames_basic` 的 `Path-first` 包装

类型：

- `channel_wrapper`

原因：

- 没有改战斗核心
- 没有改通用成长真相
- 只是在 CrazyGames Basic 下改变首页主入口、锁定场地 CTA、Quick/Path 引导关系

### Google H5 奖励广告接入

类型：

- `platform_integration`

原因：

- 依赖平台广告规则
- 依赖构建环境变量
- 依赖渠道专属 build/check

### 新增一个通用陀螺

类型：

- 通常是 `content_only`
- 如果它改变核心经济或成长曲线，也可能升级为 `core_gameplay`

## 结论

以后不是“先想到改哪里”，而是：

1. 先分类
2. 再决定改母体、共享层、还是渠道层
3. 最后走对应验证链

这一步是控制多渠道维护成本的第一道门。
